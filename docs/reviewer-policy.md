# 审核 Agent 流转策略

## 1. reviewer 的实际作用

reviewer 是质量闸门，不只是评价文案。它负责：

1. 逐条审核 copywriter 产出的内容。
2. 将单条审核状态与反馈写回内容排期表。
3. 计算项目级审核通过率。
4. 写回审核总评、审核阈值、红线风险。
5. 影响 Orchestrator 是否触发返工重试。

## 2. 项目类型如何决定阈值

项目类型来自项目主表的结构化字段 `project_type`，通常由 Brief 创建阶段或人工填表时确定，不由 reviewer 临时猜。

当前阈值配置在 `config.py`：

```python
REVIEW_THRESHOLDS_BY_PROJECT_TYPE = {
    "电商大促": 0.6,
    "日常运营": 0.6,
    "新品发布": 0.7,
    "品牌传播": 0.7,
    "母婴": 0.8,
    "医疗健康": 0.9,
}
```

如果项目类型缺失或未配置，则使用：

```python
REVIEW_PASS_THRESHOLD_DEFAULT = 0.6
```

## 3. 阈值起什么作用

审核通过率达到阈值且未命中红线风险时，项目继续进入 project_manager。

审核通过率低于阈值，或审核总评命中红线关键词时，Orchestrator 触发 copywriter 返工与 reviewer 复审。

## 4. 红线一票否决

即使通过率达标，只要审核总评中命中红线关键词，也会触发返工。默认红线关键词：

```text
严重合规风险, 虚假宣传, 绝对化用语, 医疗化表述, 编造数据, 事实错误, 严重不适配
```

可通过环境变量覆盖：

```bash
REVIEW_RED_FLAG_KEYWORDS=严重合规风险,虚假宣传,医疗化表述
```

## 5. 重试次数配置

默认最多重试 2 次：

```bash
REVIEW_MAX_RETRIES=2
```

每次重试都会执行：

```text
copywriter 返工 -> reviewer 复审
```

超过次数后不再卡死流程，而是继续进入 project_manager，由项目经理做最终交付判断。

## 6. 写回字段

项目主表新增/使用以下审核策略字段：

| 字段 | 含义 |
| --- | --- |
| 审核通过率 | reviewer 计算出的通过条数 / 总条数 |
| 审核阈值 | 当前项目类型对应的通过率门槛 |
| 审核红线风险 | 是否存在红线风险，建议值：无 / 存在 |

## 7. 推荐解释话术

reviewer 是系统里的质量闸门。它先基于规则库逐条审核内容，再计算项目级通过率。编排器根据项目类型选择不同审核阈值：例如电商大促默认 60%，品牌传播和新品发布默认 70%，母婴/医疗健康更高。同时增加红线一票否决机制，防止通过率达标但存在严重合规问题的内容继续流转。

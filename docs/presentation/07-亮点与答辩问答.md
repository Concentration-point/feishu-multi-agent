# 亮点与答辩问答

## 项目亮点

### 1. 角色化协作，不是单 Agent
很多项目只是一个模型接一个 prompt。本项目把真实团队的角色拆分为 6 个独立 Agent，各有人格（soul.md）、工具白名单和行为约束。新增角色 = 新建目录 + soul.md，零代码改动。
- **技术实现**：`agents/base.py` 的 `BaseAgent` 引擎 + `agents/{role_id}/soul.md` 配置驱动

### 2. 自研 ReAct 引擎，零框架依赖
不依赖 LangChain / AutoGen 等第三方 Agent 框架，基于 function calling + ReAct 循环自建引擎。好处是：完全可控、可解释、可调试，每一步工具调用都有完整日志。
- **技术实现**：`agents/base.py:_react_loop` — 迭代式 think → tool_call → observe → think，支持最大迭代控制和空 turn 防护

### 3. 全链路业务闭环，不是一次性输出
从 Brief 解读、人审门禁、联网调研策略、逐平台文案、红线审核+返工、排期交付，到经验蒸馏和知识沉淀，覆盖完整业务链。
- **技术实现**：`orchestrator.py` 状态驱动动态路由，`config.py:ROUTE_TABLE` 查表决定下一步

### 4. 动态路由状态机
流水线不是固定序列，而是根据项目当前状态查路由表动态决定走哪个 Agent。可从任意中间状态恢复（如人审超时挂起后，下次触发直接恢复到人审环节）。
- **技术实现**：`config.py:ROUTE_TABLE` 定义 `{状态 → 角色}` 映射，`orchestrator.py:_resolve_next_role` 查表

### 5. 文案 Fan-out 并行 + 完成度保障
Copywriter 阶段按平台拆分多个子 Agent 并行执行（信号量控制并发），完成后进行完成度检查，对空稿行定向重试，确保零遗漏。
- **技术实现**：`orchestrator.py:_run_copywriter_fan_out` — asyncio.Semaphore + 完成度扫描 + 定向补写

### 6. 三层记忆 + 经验自进化
L0 工作记忆（进程内）→ L1 项目记忆（Bitable）→ L2 经验池（Bitable + Wiki 双写）。每轮流水线结束后，Agent 自省蒸馏经验卡片 → 置信度四维打分 → 阈值过滤 → 去重合并 → 落盘。后续项目自动检索复用。
- **技术实现**：`memory/experience.py:ExperienceManager` — save → dedup → merge → optimize_bucket

### 7. 飞书深度耦合，不是纯文本对话
项目直接落在飞书多维表格（数据源）、IM（人审+广播）、知识空间（交付文档+知识库）上。19 个工具覆盖 Bitable CRUD、IM 消息、Wiki 读写、联网搜索等能力。
- **技术实现**：`feishu/` 模块 + `tools/` 19 个注册工具 + 断路器模式（`tools/__init__.py:ToolRegistry`）

### 8. 人审门禁 + 审核返工双闭环
人审门禁：AM 产出后通过飞书卡片征求人类审批，支持超时挂起/恢复/连续超时终止。审核返工：不达标自动回退到撰写阶段，由主循环路由自然驱动重试（最多 2 次），非递归。
- **技术实现**：`orchestrator.py:_enter_human_review_gate` + `_handle_reviewer_retries`

### 9. 跨项目数据分析（对应参赛模块 4）
数据分析师作为独立 Agent，从项目表+内容表+经验表三张多维表格拉取全量数据，生成运营周报、数据洞察、决策建议三种报告，通过飞书群聊推送。
- **技术实现**：`agents/data_analyst/soul.md` + `tools/query_project_stats.py` + `tools/generate_report_doc.py` + `tools/send_report.py`

### 10. 实时可观测 Dashboard
基于 SSE 事件流的 React 19 Dashboard，支持：角色切换视图、工具调用侧边栏、DAG 流程图、工具健康面板、成本统计条、历史回放。
- **技术实现**：`dashboard/react-app/` — Zustand 状态管理 + Framer Motion 动画 + SSE 实时推送

## 常见问题

### Q1：为什么不用现成 Agent 框架？
A：项目目标之一是证明，基于 function calling + 配置驱动，也能搭出清晰、可控、可解释的 Agent 系统。自建引擎的优势：完全可控的迭代逻辑、透明的工具调用链、灵活的 soul.md 配置、零外部框架依赖。

### Q2：为什么知识库采用本地优先？
A：Agent 检索需要高频低延迟访问，本地 .md 文件是最快的。飞书知识空间作为人的查看界面和长期存档，通过后台定时同步保持一致。

### Q3：如何避免低质量经验污染经验池？
A：四层防线 — ① Hook 蒸馏时 LLM 自省评估 ② 四维置信度打分（通过率 40% + 任务完成 30% + 无返工 20% + 知识引用 10%）③ 阈值过滤（默认 ≥ 0.7）④ 桶级去重合并（指纹匹配 + LLM 合并）。

### Q4：为什么要设计审核返工？
A：真实内容生产不是一次生成就结束。审核→反馈→修改是业务流程中不可省略的一环。系统通过状态回退（审核中→撰写中）让主循环路由自然驱动重试，避免递归嵌套。

### Q5：数据分析师和流水线是什么关系？
A：数据分析师独立于项目流水线，不处理单个项目，而是跨项目汇总三张多维表格的业务数据，生成统计报告和决策建议。可通过 CLI（`python main.py report`）、Dashboard API 或定时任务触发。

### Q6：经验进化的效果怎么量化？
A：每轮流水线结束时，终端输出经验进化统计：蒸馏产出 N 条 → 打分通过 N 条 → 去重合并 N 组 → 最终沉淀 N 条。后续项目中，策略师和文案会通过 `get_experience` 工具检索历史经验，决策时引用。Dashboard 的事件流也会实时展示 `experience.distilled` / `experience.scored` / `experience.saved` 等事件.

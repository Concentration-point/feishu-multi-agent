# 知识库架构设计（v1）

> 目标：把当前测试型知识堆，重构为可供多 Agent 检索、项目沉淀、审核判断与后续 Router/Planner 使用的正式知识层。

---

## 1. 设计目标

新知识库必须同时满足四件事：

1. **人能看懂**：不是测试废稿堆，不靠临时命名维持秩序。
2. **Agent 能检索**：检索结果优先命中正式、稳定、可复用知识，而不是垃圾内容。
3. **项目能沉淀**：每个项目结束后，能从执行记录中抽取出复盘与经验。
4. **支持未来分流决策**：后续 Router / Planner 需要把知识层当成决策输入，而不只是结果存档。

---

## 2. 总体原则

### 2.1 企业知识与项目知识分开
- 企业知识：长期稳定，跨项目复用。
- 项目知识：具体客户、具体项目、具体执行过程。

### 2.2 正式区与待整理区分开
自动化产物不直接进入正式知识区。先进入收件箱，经质量门槛或人工确认后再升格。

### 2.3 模板与内容分开
模板是标准件，内容是实际产出。两者不能混放。

### 2.4 客户档案与项目档案分开
客户是长期实体，项目是一次执行。不要混成一份文档。

### 2.5 经验必须可溯源
每条经验都必须带来源项目、来源运行、来源阶段、审核状态、置信度。

## 3. 顶层目录结构

建议飞书知识库与本地知识目录统一采用以下顶层结构：

```text
01_企业底座
02_服务方法论
03_行业知识
04_平台打法
05_标准模板
06_客户档案
07_项目档案
08_项目执行记录
09_项目复盘
10_经验沉淀
11_待整理收件箱
```

### 3.1 01_企业底座
放长期稳定的企业级信息：
- 企业定位
- 服务边界
- 核心服务能力
- 典型客户类型
- 风险边界

### 3.2 02_服务方法论
放内容服务的操作系统：
- 内容生产主流程
- Brief 解读规则
- 策略设计原则
- 审核规则与风险边界
- 人审规则与超时策略

### 3.3 03_行业知识
按行业沉淀长期知识：
- 行业用户特征
- 常见打法
- 典型雷区
- 案例抽象

### 3.4 04_平台打法
按平台沉淀渠道知识：
- 小红书打法
- 公众号打法
- 抖音打法
- 平台风险点
- 内容结构偏好

### 3.5 05_标准模板
放所有标准模板：
- Brief 解读模板
- 策略模板
- 文案模板
- 审核模板
- 复盘模板
- 经验卡模板

### 3.6 06_客户档案
每个客户一份长期档案：
- 行业
- 品牌定位
- 主要产品/服务
- 目标用户
- 审核偏好
- 风险禁区
- 历史合作项目

### 3.7 07_项目档案
每个项目一份主档案：
- 项目概览
- Brief 摘要
- 目标渠道
- KPI / 目标
- 时间范围
- 当前状态
- 关键策略摘要
- 交付摘要
- 关联执行记录 / 复盘 / 经验

### 3.8 08_项目执行记录
放项目过程材料：
- Brief 解读结果
- 策略方案
- 样稿
- 全量内容
- 审核记录
- 返工记录
- 交付记录

### 3.9 09_项目复盘
放项目完成后的总结：
- 做对了什么
- 做错了什么
- 根因
- 下次建议
- 值得沉淀的经验候选

### 3.10 10_经验沉淀
放正式可复用的经验卡：
- 场景
- 策略
- 结果
- 教训
- 适用角色
- 来源项目
- 来源运行
- 来源阶段
- 审核状态
- 置信度

### 3.11 11_待整理收件箱
放自动化新产物与未验证内容：
- 自动生成经验候选
- 待归类知识片段
- 临时测试内容
- 待清洗项目材料

## 4. 四层记忆模型

为了让知识库真正服务 Agent，建议把整套结构理解为四层记忆：

### 4.1 底座记忆
对应：`01~05`

回答：**我们是谁、我们怎么做事。**

特点：
- 稳定
- 低频修改
- 高复用
- 不允许普通自动化任意覆盖

### 4.2 关系记忆
对应：`06~07`

回答：**这次是谁、这次项目要干什么。**

特点：
- 半稳定
- 允许补充事实项
- 不建议频繁自动重写整篇文档

### 4.3 过程记忆
对应：`08`

回答：**这次项目具体怎么跑出来。**

特点：
- 高频新增
- 噪声大
- 适合自动写入

### 4.4 抽象记忆
对应：`09~11`

回答：**以后遇到类似情况应该记住什么。**

特点：
- 价值最高
- 最容易被污染
- 必须有质量门槛与分流机制

- 适用角色

- 来源阶段
- 审核状态
- 置信度

## 6. 写入规则

### 6.1 项目执行材料写入 08_项目执行记录
- Brief 解读
- 策略方案
- 样稿
- 审核记录
- 返工记录
- 交付说明

### 6.2 自动经验默认进入 11_待整理收件箱
不直接进入正式经验区。

### 6.3 正式经验区入库条件
必须至少满足：
- 结构完整
- 有明确场景
- 有具体策略
- 有结果
- 有教训
- 有来源信息
- 置信度达标
- 非测试垃圾

### 6.4 底座区禁止自动覆盖
以下区域默认不允许普通自动化任意重写：
- 企业底座
- 服务方法论
- 平台打法
- 标准模板

### 6.5 客户档案更新要克制
允许补充：
- 审核偏好
- 历史合作事实
- 风险禁区

不建议自动大段覆盖客户档案主内容。

## 8. 知识状态机制（建议）

除了分类，建议每篇知识增加状态：

- `seed`：自动生成、未验证
- `verified`：已验证、可用
- `canonical`：权威版本、正式标准
- `deprecated`：过时，不再推荐

这样可以避免“所有写进知识库的内容都被同等对待”。

## 9. 重建施工顺序

### Phase 0：冻结旧库
- 停止旧结构继续作为正式写入目标
- 将旧测试内容视为待淘汰资产

### Phase 1：创建新骨架
- 建立 01~11 顶层目录
- 统一命名顺序

### Phase 2：优先补底座文档
至少先创建：
- 企业定位与服务边界
- 内容生产主流程
- Brief 解读规则
- 审核规则与风险边界
- 人审规则与超时策略
- 小红书 / 公众号 / 抖音平台打法
- Brief / 策略 / 文案 / 审核 / 复盘 / 经验卡模板

### Phase 3：建立客户 / 项目双档案制度
- 为重要客户建立客户档案
- 为正式项目建立项目档案

### Phase 4：迁移有价值资产
只迁移有代表性的客户、项目、经验，不平移测试垃圾。

### Phase 5：接入自动化写入规则
- 项目执行材料 -> `08_项目执行记录`
- 自动经验候选 -> `11_待整理收件箱`
- 合格经验 -> `10_经验沉淀`

---

## 10. 最小落地包

如果只做第一轮，建议优先完成这五件事：

1. 创建 01~11 顶层目录
2. 补齐 10~15 篇底座文档
3. 定义客户档案 / 项目档案模板
4. 把自动沉淀默认写入 `11_待整理收件箱`
5. 停止旧测试知识继续写入

---

## 11. 结论

这次重建，不是在整理文档，而是在给系统重建长期记忆结构。

真正值钱的不是“删旧库”，而是：
- 明确知识分层
- 建立正式区与收件箱
- 控制自动写入边界
- 让知识能支撑未来的多链路分流与决策

做对这一步，后面的 Router / Planner、项目分型、样稿优先链路，才有真正的认知基础。

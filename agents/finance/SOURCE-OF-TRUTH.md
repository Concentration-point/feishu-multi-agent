# Finance Source of Truth

## 结论
从现在开始，**飞书多维表格是 finance 的唯一正式账本（single source of truth）**。

- 正式账本：飞书多维表格
  - app_token: `DTBKbMBRcaO9jHsY99ycAc3unid`
  - table_id: `tblX8Jop5niKoOK9`
- 本地 JSONL：降级为本地链路产物 / 调试文件 / 备份参考 / 审计材料
- 所有对外 finance 结论，默认以飞书多维表格查询结果为准

## 为什么这样定
当前实际使用中：
- 老大日常查看与核对的是飞书账本
- 真实已落账数据已经写进飞书
- 本地 JSONL 与飞书并未自动同步
- 若继续保留双正式源，主管层总结极易拿错账本，导致“账本明明有，复盘却说空白”的低级翻车

因此必须收口：

> **飞书是正式真相，本地文件不是正式真相。**

## 角色分工
### 飞书多维表格（正式层）
负责：
- 正式消费记录
- 正式状态判断
- 对外回执、复盘、日报、周报、月报、年报
- 所有“已记 / 已入账 / 已落实 / 漏记 / 空白 / 未落实”等正式结论依据

### 本地 JSONL（辅助层）
负责：
- 本地调试
- 执行链路测试
- 临时缓存
- 审计/修复脚本输入输出
- 离线分析草稿

本地 JSONL **不得单独作为正式对外依据**。

## 执行规范
### 1. 正式记账
当用户主动发来金额相关图片/金额相关信息并进入记账流程时：
- 正式落账以飞书多维表格写入成功为准
- 只有飞书写入成功并回读可见后，才能说“已记 / 已入账”
- 若本地 JSONL 有同步动作，也只能算附属动作，不改变正式状态判断

### 2. 正式复盘/总结
任何 finance 复盘、日报、周报、月报、异常说明：
- 必须先查询飞书多维表格
- 先给飞书账本数字，再下结论
- 若飞书与本地 JSONL 冲突：
  - 对外口径按飞书
  - 对内记录“本地链路未同步”

### 3. 状态词约束
以下词汇默认视为正式状态词：
- 已记
- 已入账
- 已落实
- 漏记
- 空白
- 未落实
- 已补完
- 明天补账

只要使用这些词：
- 必须基于飞书正式账本现状
- 不得基于旧聊天或本地缓存拍脑袋判断

### 4. 例外情况
只有在以下场景，才允许本地 JSONL 单独被引用：
- 明确在做本地调试
- 明确在做 ledger repair / audit
- 明确在讨论本地执行链路，不是正式财务状态

若是这种场景，必须显式写明：
- “以下为本地账本视角，不代表飞书正式账本现状。”

## 对主管层的硬要求
finance supervisor 在输出前必须先确认：
- 我查的是不是飞书主账本？
- 我说的数字是不是来自飞书？
- 我现在说的是正式状态，还是本地调试状态？

答不清，就不准发。

## builder 约束
正式 finance 汇总与复盘，后续默认应先经过 `finance-summary-builder-spec.md` 定义的 builder 结构化输出，再由 supervisor 决定如何表达。

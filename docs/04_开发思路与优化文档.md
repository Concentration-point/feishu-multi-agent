## 1 · 设计理念与架构哲学

### 1.1 顶层心智模型：**「飞书 = 共享黑板」+「soul.md = 一切人格」+「Orchestrator = 状态机」**

整个系统不是把多 Agent 框架拼起来，而是用**三件最基础的东西**把营销业务链做成可执行系统：

| 心智模型 | 对应实现 | 关键引用 |
|:---|:---|:---|
| **飞书 = 共享黑板** | 项目主表 / 内容排期表 / 经验池表是事实源，多角色读写同一行 Bitable 协作；没有内部 RPC、没有 Agent 间消息总线 | `memory/project.py`、`config.py:111-159`（FIELD_MAP_*） |
| **soul.md = 一切人格** | 角色描述、工具白名单、最大迭代数、平台补丁，全部配置在 Markdown frontmatter；新增角色零代码改动 | `agents/base.py:147-220`（`SoulConfig` + `parse_soul`） |
| **Orchestrator = 状态机** | 不固定流水线序列，每完成一个角色就**回头读一次状态**再决定下一个角色，任意中间状态都能恢复 | `config.py:223-236`（`ROUTE_TABLE`）、`orchestrator.py:99-104`（`_resolve_next_role`） |

这三件事配齐后，几乎所有别的设计都是它们的自然推论。

### 1.2 五条贯穿全栈的设计原则

1. **配置驱动，零代码扩展**
   新增角色 = 新建 `agents/{role_id}/soul.md`；新增平台分化 = 在 `agents/{role_id}/platforms/` 放 `.md`；新增状态 = 改 `config.ROUTE_TABLE`。
   反例：没有任何角色相关的硬编码 if/else 散落在 Python 代码里（`agents/base.py:243-316` 的平台补丁是「扫描式发现」，连白名单都不需要）。

2. **本地优先 + 后台同步**
   Agent 检索走本地 `grep`（毫秒级，见 `tools/search_knowledge.py`），异步线程把变更推到飞书知识空间（`sync/wiki_sync.py:94-103`）。
   反例：从不在 Agent 主链路上走飞书 API 拉知识，避免 API 限频和延迟传染主流程。

3. **单向同步 + 严格白名单**
   01-06 是人维护、只下行；07-10 是 Agent 产出、只上行；11 是脏缓冲、不出站；`references/` 是对标库、永不出站。
   实现：`sync/wiki_sync.py:34-92`（`_DEFAULT_UPLOAD_INCLUDE_DIRS` + `_is_excluded`）。
   设计意图：**人类层和 Agent 层任何时候都能溯源谁是真值**，避免互相覆盖。

4. **经验可溯源**
   每条沉淀的经验都带 `source_project / source_run / source_stage / review_status` 四元组（`memory/experience.py:262-271`、`orchestrator.py:1287-1292`）。
   出问题的经验能反查回某次 run、某个角色的具体执行；治理可观测。

5. **失败要被「看到」而不是被「吞掉」**
   工具异常转字符串返回给 LLM 让 ReAct 循环能自纠（`tools/__init__.py:153-173`）；阶段失败回写到 `agent_error_log` 字段让运营侧能在表格里感知（`orchestrator.py:245-248`）；阶段超时硬中止（`orchestrator.py:373-378`）；工具熔断（`tools/__init__.py:99-117`）；Wiki 同步失败保留 dirty 等下次（`sync/wiki_sync.py:142-158`）。
   反例：禁止使用裸 `except: pass`，所有捕获都至少 `logger.warning` + 状态可观测。

### 1.3 设计模式落点

| 模式 | 应用位置 | 用意 |
|:---|:---|:---|
| **策略模式** | `_ROLE_REFLECT_PROMPTS`（`agents/base.py:130-134`） | 每个角色用不同自省 prompt，主循环代码不变 |
| **注册表 + 自动发现** | `tools/__init__.py:58-79`（`ToolRegistry._scan`） | 新增工具 = 在 `tools/` 放新 `.py`，无需改注册表 |
| **熔断器** | `tools/__init__.py:99-117` | 防止某个外部依赖（飞书 / Tavily）抖动拖垮整条流水线 |
| **状态机 + 路由表** | `config.py:223-236` + `orchestrator.py:161-274` | 用「状态 → 角色」的纯数据结构替代分支 if/else |
| **观察者（事件总线）** | `dashboard/event_bus.py` + `BaseAgent._publish`（`agents/base.py:492-525`） | Agent / Orchestrator 不知道 Dashboard 存在，只发事件 |
| **装饰器（实质上的）** | `tools/__init__.py:118-173`（`call_tool` 包裹熔断 + 日志 + JSONL 持久化） | 把横切关注点（监控、限流）从工具实现中剥离 |

## 2 · 实现思路与技术选型

### 2.1 整体实现思路：从「能跑通」反推到「能治理」

走的路线：**业务决定接口 → 接口决定状态 → 状态决定角色 → 角色决定工具 → 工具决定外部依赖**。

```
飞书表字段（业务真值）
  ↓ FIELD_MAP_*（config.py）
Bitable 行结构（数据契约）
  ↓ ProjectMemory / ContentMemory（数据访问层）
状态枚举（STATUS_*）
  ↓ ROUTE_TABLE（动态路由）
六个角色（soul.md）
  ↓ tools 白名单
21 个工具
  ↓ 直调 OpenAPI
飞书 API / LLM API / Web 抓取
```

任何一层都可以**只改自己这层**就完成扩展：加新平台 → 改 soul 补丁；加新审核维度 → 改 reviewer soul + 加字段映射；加新外部依赖 → 加新工具文件。

### 2.2 关键技术决策

| 决策 | 选择 | 不选什么 | 理由 |
|:---|:---|:---|:---|
| Agent 框架 | **自研 ReAct 循环**（`agents/base.py:734-816`） | LangChain / AutoGen / CrewAI | 框架的抽象会绑死 prompt 装配方式与中间状态结构，soul.md 配置驱动需要完全自定的 prompt 拼接顺序（`_build_system_prompt`，`agents/base.py:1036-1079`）。自研 200 行 ReAct = 框架价值的 80%，但保留 100% 控制权。 |
| Agent 间通信 | **Bitable 行作为共享黑板** | 内部消息队列 / RPC | 业务方本来就要在飞书查看进度，让 Bitable 做事实源等于用户界面 + 状态存储一份；没有「内部状态 vs 外部展示」漂移。 |
| 知识检索 | **本地 .md + 多关键词 grep** | 向量数据库（FAISS / Milvus） | 知识量级在百~千篇 Markdown，grep 毫秒级；上向量库带来：embedding 模型依赖 + 重建索引 + chunk 切分错位问题，**ROI 不正**。如果未来到 10w 篇再换。 |
| 飞书集成 | **直调 OpenAPI（httpx）** | 飞书官方 SDK | SDK 是同步的，不支持 asyncio；OpenAPI 直调反而更轻、可控（见 `feishu/bitable.py` 的全局并发闸门设计）。 |
| LLM 协议 | **OpenAI Chat Completions + function calling** | 各家 SDK 各异、Anthropic / 智谱 / DeepSeek 各写一套 | OpenAI 协议事实上已成 LLM 调用层的「TCP/IP」，通过 `LLM_BASE_URL` 切换厂商（`config.py:78`）。 |
| YAML 解析 | **手写极简解析**（`agents/base.py:169-220`） | PyYAML 依赖 | soul.md frontmatter 字段固定（name/role_id/tools/max_iterations），手写 30 行解析比拉一个 PyYAML 更轻；但**这是技术债**（见 6.1 节）。 |
| 前端框架 | **React 19 + Vite 8 + Zustand + Tailwind 4** | Streamlit / Gradio / Vue / Svelte | 实时 SSE 流 + 多状态管道动画用 React + Zustand 自然；Streamlit 渲染开销大、不适合实时面板。 |
| 异步模型 | **全 async/await（含飞书 API 并发闸门）** | 多线程 / 多进程 | I/O 密集（飞书 API + LLM API + 网页抓取），asyncio 单进程能撑很多并发；并发闸门见 `feishu/bitable.py`。 |

### 2.3 实现路径（怎么从 0 长到现在的样子）

代码考古能看出三个里程碑：

1. **第一阶段：跑通五角色固定序列**
   `Orchestrator.DEFAULT_PIPELINE`（`orchestrator.py:55-61`）保留至今，仅作为 Dashboard 的 stage_total 计算和兜底；这个常量是早期固定流水线的化石。

2. **第二阶段：动态路由 + 人审门禁 + 返工**
   引入 `ROUTE_TABLE` + `_resolve_next_role` + `_handle_reviewer_retries`，把固定数组升级成状态机。
   关键拐点：发现「项目可能掉线、需要恢复」「审核可能要返工」之后，固定数组写死了就再也接不住——必须状态驱动。

3. **第三阶段：经验沉淀 + 知识治理 + 实时可视化**
   引入三层记忆 + Hook 自省 + 11 层知识 + 双向同步白名单 + Dashboard SSE。
   这一阶段的代码量爆炸增长（`orchestrator.py` 已经 1390 行），是当前**主要的技术债和优化机会的来源**（见 6 节）。

## 3 · 创新点分析

### 3.1 技术创新

#### 创新 1：**「状态驱动动态路由」替代固定 pipeline 数组**

**问题**：固定流水线 `[AM → S → C → R → PM]` 写死在数组里，遇到三种情况就崩：
- 项目执行到一半进程挂了 → 重启后无法从中间恢复
- 审核驳回需要返工 → 必须在主循环里硬编码「if reviewer 不通过 → 跳回 copywriter」
- 客户在飞书直接修改了 Brief 解读字段 → 系统不知道要不要重跑

**解决方案**（`orchestrator.py:161-274`）：
```python
while step < self._max_route_steps:
    next_role = self._resolve_next_role(current_status)  # 查 ROUTE_TABLE
    if next_role is None: break                          # 终止状态
    # ... 执行 next_role ...
    current_status = await self._read_current_status()   # 每步重读状态
```

**价值**：
- 任意中间状态可恢复（`status="待人审"` 直接进入门禁恢复，跳过 AM 重跑，见 `orchestrator.py:136-150`）
- 路由表可配置（新增状态 = 改 `ROUTE_TABLE` 字典，不动主循环）
- `MAX_ROUTE_STEPS=15` 安全上限防死循环（`config.py:84`）
- 配合 `_handle_reviewer_retries` 把状态改回「撰写中」即触发返工，不需要任何分支跳转代码（`orchestrator.py:556-624`）

**对比业界**：LangGraph 等框架也有 state machine 模式，但要求把整个 graph 用代码定义；这里把 graph 等价表示为**一张字典**，配置驱动到底。

#### 创新 2：**Hook 自省蒸馏 + 置信度评分漏斗 + 桶级 LLM 合并**

**问题**：常见的「Agent 经验沉淀」要么写到一个 dump 文件（无结构）、要么直接 append 到向量库（爆炸增长）、要么靠人工筛选（不可持续）。

**解决方案**（三段式漏斗，不是单纯「写一条」）：

1. **蒸馏阶段**（`agents/base.py:932-981`，`_hook_reflect`）
   每个角色用专属 prompt 自我回顾 ReAct 历史，吐出 SAOL（Situation/Action/Outcome/Lesson）JSON 卡片。每个角色 prompt 不同（`_ACCOUNT_MANAGER_REFLECT_PROMPT` 让 AM 反思人类怎么修正、`_COPYWRITER_REFLECT_PROMPT` 让文案反思对标 vs 规则的冲突），不是一刀切。

2. **打分阶段**（`orchestrator.py:1360-1372`，`_calc_confidence`）
   ```
   confidence = 0.4 * pass_rate + 0.3 * task_completed + 0.2 * no_rework + 0.1 * knowledge_cited
   ```
   < 0.7 直接丢弃（`config.py:27` 的 `EXPERIENCE_CONFIDENCE_THRESHOLD`）。

3. **合并阶段**（`memory/experience.py:622-688`，`optimize_bucket`）
   按「角色 + 场景分类」分桶，桶内做：jaccard 相似度去重（`_lesson_similarity`，`memory/experience.py:129-136`）→ LLM 二次合并（`_merge_bucket_records`）→ 经验内容超过 200 字自动压缩到 100 字内（`_compress_lesson`）。

**关键差异**：业界常做「写入时去重」（在写入前查相似条），这里做的是**「桶级周期性重组」**——每条新经验落库后，对所属桶做一次 dedup + merge，不止防止重复，还会**主动用 LLM 把冗长经验压成可执行清单**。

**反例处理**：经验用 `applicable_roles` 扇出写多条 Bitable 记录（`memory/experience.py:198-235`），这样 reviewer 沉淀的「合规检查清单」也能被 copywriter 的 `query_top_k` 查到，跨角色复用。

#### 创新 3：**copywriter 双轨工作流 + 内嵌 HTML 注释可追溯**

**问题**：纯靠 LLM 写文案，要么爆款套路抄到违规、要么死守规则写得没质感。

**解决方案**（`agents/copywriter/soul.md:38-97`）：
- **轨道 A**：必须先调 `search_reference` 拿真实爆款，分析 hook / structure / CTA
- **轨道 B**：必须调 `search_knowledge` 至少 2 次（禁用词 + 平台规范）
- **融合规则**：「规则优先于爆款」，对标里的违规措辞用合规替代方案改写
- **可追溯**：成稿正文最顶部强制注入 HTML 注释块（`<!-- [对标参考] ... [合规自检] ... -->`），不渲染但写入 draft，审核阶段会回看 ReAct 日志验证一致性
- **代码级硬约束**：`_REQUIRED_TOOL_CALLS` 强制 copywriter 必调 `search_reference + search_knowledge`，不调就在最终输出注入合规警告（`agents/base.py:138-144`、`agents/base.py:818-828`）

**为什么这是创新**：业界常见做法是把对标和合规分别交给两个 Agent，但**两个 Agent 之间的协调成本远大于单 Agent 内部硬约束**。这里把双轨做成单 Agent 内部的「双工具栈」，在 prompt 层面强制走完，在代码层面兜底校验，在产物层面留下溯源痕迹。

#### 创新 4：**copywriter fan-out 并发 + 串行重试**

**问题**：一个项目可能同时要写「公众号 + 小红书 + 抖音」三个平台，串行写 3 次 Agent，平台互不影响却被串行挤压。

**解决方案**（`orchestrator.py:386-554`，`_run_copywriter_fanout`）：
1. 按 `row.platform` 分组
2. 每个平台一个子 Agent（共享 `record_id`，独立 `task_filter={"platform": X}`）
3. `asyncio.gather + Semaphore(5)` 并发 = 5
4. `return_exceptions=True` 捕获失败组，串行重试 1 次
5. 每个子 Agent 的 `_pending_experience` 独立收集

**精妙之处**：
- 平台补丁通过 `task_filter.platform` 自动激活对应 `agents/copywriter/platforms/{platform}.md`（`agents/base.py:253-316`），同一份 base soul + 不同平台补丁 = N 个差异化 Agent
- fan-out 子 Agent 不主动推进全局状态，由 Orchestrator 统一在所有子任务完成后兜底推进到「审核中」（`orchestrator.py:251-267`）—— 避免子 Agent 之间状态推进竞态

#### 创新 5：**11 层知识 + 双向白名单 + 升格审批**

**问题**：让 Agent 自由读写知识库会怎样？短期：到处是脏经验；长期：人类维护的内容被 Agent 覆盖、可追溯性丢失。

**解决方案**（`docs/knowledge-architecture.md` + `sync/wiki_sync.py`）：
- 11 层数字前缀 + `references/` + `_index.md`
- **方向规约**：01-06 只下行（人 → Agent），07-10 只上行（Agent → 飞书），11 不出站，`references/` 不出站
- **升格机制**：11 → 10 必须经过飞书「升格审批表」人工审批（`scripts/submit_inbox_to_review.py` → `apply_approved_promotions.py`）
- **白名单强制**：`WIKI_SYNC_UPLOAD_DIRS` 默认只放 07-10，环境变量可覆盖但默认拒绝

**为什么这是治理而不只是同步**：很多团队上 Wiki 同步是「全量双向，hash 谁新覆盖谁」——这在多 Agent 场景下灾难性，因为 Agent 写得快、人写得慢，Agent 永远是 source of truth，违反业务直觉。

### 3.2 模式创新

#### 模式 1：**人格 = 配置 + 提示 + 工具白名单 + 平台补丁**

```
agents/copywriter/
├── soul.md                  # 基础人格（YAML frontmatter + Markdown body）
└── platforms/
    ├── 小红书.md            # 平台补丁（纯 Markdown，无 frontmatter）
    ├── 公众号.md
    └── 抖音.md
```

合成规则（`agents/base.py:253-316`）：
- 扫描式发现：有 `platforms/` 目录就触发，无白名单
- platform 缺失 → 软兜底（用基础 soul + 打 `fallback_used: true` 事件给 dashboard）
- 工具权限**只在 base soul** 声明，平台补丁不能改

**为什么是模式创新**：传统多 Agent 用「prompt template + 变量替换」做差异化；这里把差异化做成**文件系统结构**，让平台补丁可以被产品 / 内容运营直接编辑，不用动 Python。

#### 模式 2：**飞书表格作为「跨角色协议」**

每张表的字段都是公开协议（`config.py:111-184`）。任意角色 / 工具 / 脚本都遵守同一份字段映射。
配套：`reviewer` 约定输出四个结构化字段（`review_pass_rate / review_red_flag / review_status / review_summary`），Orchestrator 据此决策——**reviewer 改 prompt 都不影响 Orchestrator**，只要四字段不变。

### 3.3 业务创新

#### 业务 1：**「人审门禁」嵌入流水线但不阻塞**

人审 = 把客户经理产出推到飞书群，等待群消息审批 / 修改 / 超时（`orchestrator.py:625-737`）。
- 通过 → 进策略
- 修改 → 写反馈到 `human_feedback` 字段，回退到「解读中」状态，下次触发会自动注入反馈让 AM 重写（`agents/base.py:1063-1071`）
- 超时 → 不丢数据，把状态改成「待人审」挂起，下次触发同 record_id **跳过 AM 直接进恢复门禁**

业务上的好处：客户在群里随便丢一句「修改：调性再放松点」，下一次提交就能从断点继续。

#### 业务 2：**审核四字段 + 项目类型分级阈值**

不同项目类型阈值不同（`config.py:41-48`）：
```
电商大促 0.6, 新品发布 0.7, 母婴 0.8, 医疗健康 0.9
```
配合 `REVIEW_RED_FLAG_KEYWORDS`（`config.py:34-39`）：命中红线一票否决，不走返工。
这是把行业合规等级直接表示为配置常量。

#### 业务 3：**交付文档自动生成（面向客户）**

流水线完成后自动生成飞书云文档（`orchestrator.py:1038-1201`）：
- 自动定位 / 创建「项目交付文档」父节点（进程级缓存，单次创建）

### 4.2 性能瓶颈识别

#### 瓶颈 1：**经验合并的 LLM 成本会随项目数线性增长**

每完成一个项目，每个角色 + 每个场景分类的桶都可能触发 `optimize_bucket`（`memory/experience.py:622-688`），其中 `_merge_bucket_records` 是一次完整 LLM 调用。
- 角色 6 个 × 场景 4 个 = 24 个桶
- 假设 30% 桶超过 `EXPERIENCE_MAX_PER_CATEGORY=3` 触发合并 → 每个项目末尾 ~7 次 LLM 调用
- 100 个项目跑下来 ≈ 700 次合并调用，按 GPT-4o 估算 $5-15

**优化方向**：
- 方案 A（低成本）：把合并改成「批延迟」，攒 N 个项目跑一次桶级 dedup，而不是每个项目都跑
- 方案 B（高质量）：用 embedding 相似度替代 jaccard tokenize，先在向量空间做粗筛，只对相似度 > 0.8 的对调 LLM 合并
- 方案 C（治理）：把 `optimize_bucket` 从 Orchestrator 主路径移到独立的后台 cron 任务，主流水线不再阻塞合并

#### 瓶颈 2：**Bitable 写入的并发冲突**

`copywriter` fan-out 启了 5 个并发子 Agent，每个子 Agent 都会调 `write_content` 写多行。飞书 Bitable 单表写频率 ~5 QPS，5 个并发 × 每个写 N 行很容易触发限频。
现状靠「全局并发闸门」（`feishu/bitable.py` 的 Semaphore）排队，**等价于把并行降回串行**，fan-out 的并发收益被吃掉。

**优化方向**：
- 用 Bitable 的 `batch_create` / `batch_update`，把同一子 Agent 的多行写入合并成 1 次 API 调用
- 在 `BatchAgentOps` 层缓冲 N 秒内的写入，统一 flush
- 现在 `tools/batch_create_content.py` 已经存在但只有策略师用，需要让 copywriter 也走 batch

#### 瓶颈 3：**Dashboard SSE 总线进程内单实例**

`dashboard/event_bus.py` 是进程内 EventBus，**多实例部署会丢事件**：用户接的是 instance A 但 Orchestrator 跑在 instance B，A 的 SSE 收不到 B 的事件。
当前架构假设单进程，但 `python main.py serve` 上 K8s 多副本就会爆。

**优化方向**：换 Redis Streams / Kafka / NATS，或在 nginx 层做 sticky session 让同一 record_id 总打到同一实例。

#### 瓶颈 4：**`.sync_state.json` 单文件 + 单线程**

`sync/wiki_sync.py:167-177` 每次完整读写 JSON 文件。
- 文件越大写入越慢
- 多实例同时跑同步会互相覆盖
- 没有锁，崩溃时可能写出半个 JSON

**优化方向**：
- 短期：写入用临时文件 + rename 原子替换
- 中期：state 改 SQLite（自带锁 + 增量更新）
- 长期：上 Redis 做分布式 dirty 队列

#### 瓶颈 5：**经验合并后置信度直接继承 max_confidence**

`memory/experience.py:673-685` 用合并前的 `max_confidence` 给合并后的卡片打分，等于「合并永远不降分」。
理论上合并的产物质量未必优于原始最高分（LLM 合并可能漏要点），但分数照旧。

**优化方向**：合并产物经过一次「合并质量自检 prompt」，置信度按是否覆盖关键要点重新评分。

#### 瓶颈 6：**知识检索是线性 grep**

`tools/search_knowledge.py` 在 `knowledge/` 全树 rglob `*.md` 然后 grep 关键词。
- 现状：百~千篇 md，grep 5-50ms
- 1w 篇 md → 0.5-2s 已经会被感知
- 10w 篇 → 不可用

**优化方向**：
- 短期：内存缓存上次扫描的文件列表 + mtime，未变就不重扫
- 中期：建本地 BM25 / Whoosh 倒排索引（仍纯 Python）
- 长期：上 Meilisearch / Tantivy / 本地 chroma（仅在量级真到的时候）

## 5 · 可扩展性设计

### 5.1 横向扩展（功能横向）

#### 5.1.1 加角色

零代码：
1. 新建 `agents/{new_role}/soul.md`
2. frontmatter 声明 `name / role_id / description / tools / max_iterations`
3. 在 `config.ROLE_NAMES` 加中文名
4. 在 `config.ROUTE_TABLE` 加状态映射（如果加入主流水线）

例：`agents/data_analyst/` 是后加的独立 Agent，不在主流水线，CLI `python main.py report` 直接调用（参考 `main.py`）。

#### 5.1.2 加平台补丁

零代码：
1. 在 `agents/copywriter/platforms/` 放 `{platform}.md`
2. 内容行的 `target_platform` 字段填新平台名
3. fan-out 自动按 platform 分组并加载补丁（`agents/base.py:248-316` 是扫描式发现）

#### 5.1.3 加工具

零代码：
1. 在 `tools/` 放 `{tool_name}.py`
2. 导出 `SCHEMA`（OpenAI function calling 格式）+ `async def execute(params, context)`
3. 在需要使用的 soul.md 的 `tools:` 列表里加该工具名

`ToolRegistry._scan` 会自动发现注册（`tools/__init__.py:58-79`）。

#### 5.1.4 加状态 / 路由分支

低代码：
1. 在 `config.py` 加 `STATUS_*` 常量
2. 在 `VALID_STATUSES` 加新状态
3. 在 `ROUTE_TABLE` 加状态 → 角色映射

特殊路由如「人审门禁」用占位符 `__human_review_gate__`，由 Orchestrator 主循环识别（`orchestrator.py:172-180`）。

### 5.2 纵向扩展（深度增强）

#### 5.2.1 多模态内容生成

当前 copywriter 只产出文本。要扩到「文 + 图 + 视频脚本 + 视频生成」：
- 加 `tools/generate_image.py`（接 DALL·E / 豆包文生图 / Midjourney API）
- 在 `agents/copywriter/platforms/小红书.md` 补丁里加封面图生成约束
- `write_content` 字段映射新增 `cover_image_url` 字段

#### 5.2.2 跨项目联动分析

当前 `data_analyst` 是独立 Agent，可以扩展成：
- 周报 → 自动识别共性问题 → 自动生成「共性问题档案」推到 `09_项目复盘/`
- 多项目对比 → 找出 ROI 最高的内容矩阵模式 → 反向沉淀到 `10_经验沉淀/`

#### 5.2.3 客户自助配置

当前所有阈值、字段映射、ROUTE_TABLE 是 Python 常量。要支持「不同客户用不同阈值」：
- 在 `06_客户档案/` 下放每客户配置 yaml
- Orchestrator 启动时按 `client_name` 加载客户级 override
- `REVIEW_THRESHOLDS_BY_PROJECT_TYPE` 的字典查询支持「客户级 → 项目级 → 默认」三级回退

### 5.3 扩展限制

#### 5.3.1 飞书 Bitable 单表 5 万行硬上限

飞书多维表格单表限制约 5 万行。L1 / L2 都落 Bitable，长期累积会撞墙。
- 项目主表：1 项目 1 行 → 50000 个项目 ≈ 1.5 年（按每天 100 项目）
- 内容排期表：1 项目 N 行 → 5000-10000 项目就到上限
- 经验池：每个项目 6 角色 × 4 类 = 24 条 → 2000 项目就到上限

**应对**：
- 短期：加表分片（按客户 / 按月）
- 长期：内容排期表迁移到 PostgreSQL，飞书表只做「最近 N 天的视图」

#### 5.3.2 LLM API 速率与成本

依赖 OpenAI / DeepSeek 的速率配额。瓶颈：
- copywriter fan-out 5 并发 + 14 max_iterations + 每轮 1 次 LLM = 单项目 ~70 次 LLM 调用
- 50 项目并发 = 3500 次/分钟 → 大概率撞 RateLimit

**应对**：
- 多 LLM 厂商分流（`LLM_BASE_URL` 已经支持，但需要 Agent 间配置不同）
- Token-based budget（项目级预算，超了降级到便宜模型）

#### 5.3.3 单进程 asyncio 事件循环上限

asyncio 单进程性能极限大约 1000-2000 并发协程，超过会调度开销显著。
- 当前架构默认单进程
- Webhook 高峰期（飞书事件批量推送）可能阻塞

**应对**：
- 中期：用 `multiprocessing` 把 Orchestrator 改成多进程 worker pool
- 长期：拆 webhook 接收 → 任务队列（Redis）→ worker 进程

| 测试覆盖率 ≥ 70%（核心模块 ≥ 85%） | `pytest --cov` 报告 |

### 7.2 中期（3-6 个月）：能力深化 + 跨项目联动

**目标**：让系统从「执行单项目」升级为「沉淀业务智慧」。

| 项 | 业务价值 |
|:---|:---|
| 多模态文案（图 + 视频脚本 + 封面） | 减少设计师人工接力 |

| Bitable 内容表分片（按月归档） | 突破 5 万行硬上限 |

### 7.3 长期（6-12 个月）：平台化 + 生态

**目标**：从「内部工具」演化为「Agent 平台」。

| 项 | 形态 |
|:---|:---|
| Agent 市场（plugin 系统） | 客户上传自己的 soul.md + 工具，平台化分发 |
| 跨企业经验隔离（多租户） | Bitable 多 app + 命名空间 + 审计日志 |

## 8 · 设计权衡

### 8.1 关键权衡

| 决策 | 选了什么 | 放弃了什么 | 后果 |
|:---|:---|:---|:---|
| Bitable 当事实源 | **业务可见 / 协作零成本** | 性能（5 万行硬上限 + 5 QPS 限流）、复杂查询能力 | 业务侧价值满分；技术侧需要在限流和分片上多投入 |
| 自研 ReAct 引擎 | **prompt 装配自由 / 调试简单 / 0 框架依赖** | 框架社区生态（LangChain 的 1000+ 集成） | 当前业务用不到那 1000+ 集成，但需要自己实现新模式 |
| 知识检索走 grep | **毫秒级响应 / 无外部依赖** | 语义检索能力 | 量级 < 1w 篇 .md 时绝对正确；超过则需切向量库 |
| 配置驱动（soul.md） | **新增角色零代码** | 强类型校验 / IDE 补全 | 写错 yaml 不会被编译器抓；要用启动时校验补 |
| 状态机用字典而不是代码 | **新增状态零代码** | IDE 跳转 / 静态分析 | 状态分支视图不够直观；需要靠 Dashboard 可视化补 |
| 单进程 asyncio | **简单 / 调试容易** | 高并发 / 多副本扩展 | 当前规模够用；规模化要拆任务队列 |
| 经验池每条扇出多角色写入 | **跨角色复用 / 查询命中率高** | 存储成本（同一经验存多份）/ Bitable 行数翻倍 | 接受存储 cost，换查询性能 |
| 文案双轨 + HTML 注释溯源 | **审核可追溯 / 合规可证明** | 成稿正文有「冗余」注释（虽然不渲染） | 业务侧需要溯源能力；运营接受度需要培训 |

### 8.2 隐含的假设

这些假设一旦失效，整体设计就要重审：

1. **「飞书是用户工作主场」**：客户经理、审核人都用飞书。如果未来要支持「客户用钉钉 / Slack」就得做 IM 抽象层。
2. **「Markdown 知识够用」**：当前经验都是文本。如果要存「数值指标 / A/B 测试结果」就需要结构化存储（数据库）。
3. **「Agent 间不需要直接对话」**：通过 Bitable 共享黑板已经够。但 `negotiate` 工具（已存在）暗示未来可能要 Agent 间结构化协商，那就要 Agent 间的会话状态管理。
4. **「LLM 输出可结构化」**：reviewer 输出四字段、Hook 输出 SAOL JSON。如果模型输出稳定性下降需要 retry / repair 层（现在 `_clean_llm_json` 是简单的 markdown 块剥离，遇到 JSON 错位就跪）。
5. **「单租户」**：当前没有 client_id 隔离，所有项目共享同一个 Bitable / 经验池。多客户要做命名空间。

### 8.3 不变量（哪怕重构也不能动的东西）

1. **soul.md 是人格唯一来源**——不能在 Python 代码里再加 if role_id == "X" 这种分支
2. **Bitable 是 L1 事实源**——不能把项目状态缓存到内存（重启就丢）
3. **01-06 不被 Agent 写**——白名单是底线
4. **每条经验必须可溯源**——`source_project / source_run / source_stage` 不能空

## 9 · 最佳实践（已在做的）

### 9.1 编码规范

- 中文注释 + 英文标识符（CLAUDE.md 强约定）
- 全异步 I/O；同步操作（如文件读写）放在异步函数内调
- 配置常量集中在 `config.py`，不散到业务代码
- 飞书 API 只走 `feishu/`，不直接 httpx
- 工具异常转字符串返回给 LLM，让 ReAct 可见可纠

### 9.2 工程流程

- `pytest` 离线 fixture mock 飞书 / LLM；`*_live.py` 测试要真实凭证（参考 `tests/conftest.py`）
- `scripts/check_demo_ready.py` 跑 demo 前自检环境
- `scripts/preview_wiki_sync.py` 同步前 dry-run
- `scripts/audit_wiki_local.py` 审计本地 wiki frontmatter / NUL 字符
- 每个新功能配套 `docs/` 下设计文档（`agent-contract-design.md` / `module1_module2_combo.md` 等）

### 9.3 治理规范

- 11 → 10 升格必须人工过审（`scripts/submit_inbox_to_review.py`）
- 经验置信度 < 0.7 直接丢，不进入索引
- reviewer 红线一票否决，不返工
- max_route_steps / max_iterations / stage_timeout 三层防死循环

### 9.4 可观测性（已落地的部分）

- SSE 事件总线 + React Dashboard 实时面板
- `logs/tool_calls.jsonl` 跨运行累积工具调用统计
- `agent_error_log` 字段把 Agent 异常落到 Bitable 让运营可见
- `experience.*` 7 种事件可视化经验全生命周期

# 人机协同进化 — 原子任务拆解

把"Agent 自进化"升级为"人机协同进化"，三个角色各自走向不同的外部权威经验源：
- **审核 Agent** ← 规则库 + 合规知识（客观权威）
- **文案 Agent** ← 真实博主爆款（对标学习）
- **客户经理 Agent** ← 人类专家审核（主观权威）

每个任务独立，建议按模块并行开发。下方按角色分组。

---

## 模块一：文案 Agent 的"对标学习进化"

### T1-01: 实现 search_reference 搜索工具

**目标**: 文案 Agent 撰写前能搜到真实博主的爆款内容作为对标参考

**产出文件**: `tools/search_reference.py`

**实现方案**（三选一，按可行性排序）：

- **方案 A（推荐）**：调 Bing Search API 或 SerpAPI，传入查询如"小红书 精华液 种草"，返回前 5 条结果的标题 + 摘要 + URL。注册一个免费的 SerpAPI key（每月 100 次免费）就够 Demo 用
- **方案 B**：直接爬小红书/知乎的公开搜索页，用 httpx + BeautifulSoup 解析。风险是反爬机制
- **方案 C（兜底）**：本地放一个 `knowledge/references/` 目录，预存 20-30 篇真实爆款笔记的 .md 文件（可人工从小红书复制），search_reference 内部就是 grep 这个目录

**SCHEMA**:
```json
{
  "name": "search_reference",
  "description": "搜索真实博主的爆款内容作为写作对标参考",
  "parameters": {
    "query": "搜索关键词，建议格式：平台+品类+角度",
    "platform": "小红书/抖音/公众号",
    "max_results": 5
  }
}
```

**返回格式**：
```
找到 3 篇对标参考：

1. [小红书·1.2w赞] "30岁才懂的精华液真相"
   开头：闺蜜说我皮肤变好了...（前 300 字）
   结构：个人故事 → 成分解析 → 使用对比 → CTA
   
2. ...
```

**验收标准**：
- search_reference 工具出现在 ToolRegistry 中
- 给定查询能返回至少 3 条结构化结果
- 每条结果包含标题、摘要、结构分析（可选）

**预估耗时**: 2-3h（用 SerpAPI）或 1h（用本地方案）

### T1-02: 更新 copywriter/soul.md — 强制对标流程

**目标**: 文案 Agent 撰写前必须先搜爆款，禁止闭门造车

**产出**: 修改 `agents/copywriter/soul.md`

**frontmatter 变更**：
```yaml
tools:
  - read_project
  - list_content
  - write_content
  - search_knowledge
  - search_reference    # 新增
  - update_status
  - send_message
```

**body 关键指令添加**：
```markdown
## 强制工作流

在撰写任何一条内容前，你必须：

1. 先调 `search_reference` 搜索该平台+品类的爆款内容
2. 分析 3-5 篇对标参考的共性：开头方式、结构节奏、情绪调性
3. 在撰写 draft 前，明确写出"我参考了哪些爆款的什么元素"
4. 基于这些元素创作，而不是凭空生成

**不搜对标直接动笔视为违规**。审核 Agent 会检查你的撰写过程是否走了对标流程。
```

**验收标准**：
- 文案 Agent 在 ReAct 日志中能看到 search_reference 的调用
- 成稿中能识别出对标学习的痕迹（如开头风格接近参考）

**依赖**: T1-01

**预估耗时**: 30min

### T1-03: 文案经验沉淀的 Hook prompt 定制

**目标**: 文案 Agent 的自省产出"爆款模仿套路"，不是空泛的写作心得

**产出**: 修改 `agents/base.py` 中的 _hook_reflect 方法，按 role_id 分化自省 prompt

**文案专用自省 prompt**：
```
你刚完成了一轮文案撰写。回顾你这次的对标学习过程：

1. 你搜索了哪些爆款作为参考？
2. 这些爆款的共性是什么？（开头结构/情绪/信息密度/节奏等）
3. 你在创作中复用了哪些元素？
4. 归纳出一条可以传给未来文案 Agent 的"爆款模仿套路"

输出 JSON:
{
  "situation": "某平台某品类的文案撰写",
  "reference_pattern": "爆款的共性规律",
  "action": "你如何模仿",
  "outcome": "成稿质量",
  "lesson": "下次撰写同类内容时，应该先..., 然后..., 避免...",
  "category": "电商大促|新品发布|品牌传播|日常运营",
  "applicable_roles": ["copywriter"]
}
```

**验收标准**：
- 蒸馏出的经验 JSON 包含 `reference_pattern` 字段
- `lesson` 字段具体到可复用的写作套路（不是"注意语言风格"这种废话）

**依赖**: T1-01, T1-02

**预估耗时**: 1h

## 模块二：审核 Agent 的"规则驱动进化"

### T2-01: 建立合规规则知识库

**目标**: 审核 Agent 有客观的审核依据，不是 LLM 主观判断

**产出**: `knowledge/raw/rules/` 目录下的多个规则文件

**必备文件**：

**knowledge/raw/rules/广告法禁用词.md**
- 列出中国广告法禁用的绝对化用语：最、第一、最佳、顶级、绝对、国家级、最高级、独家、唯一等
- 化妆品禁用功效词：治疗、根治、消除、抗菌、杀菌、医疗等
- 保健品禁用词：预防、治愈、诊断等
- 每类下列 30+ 具体词汇
- 附带合规替代方案（如"最佳" → "推荐"、"顶级" → "高端"）

**knowledge/raw/rules/平台规范.md**
- 小红书：笔记字数限制、禁用内容、标签数量、图片规范
- 抖音：脚本时长、口播敏感词、视频内容红线
- 公众号：标题长度、原创要求、违规词汇
- 微博：话题规则、字数限制

**knowledge/raw/rules/品牌调性检查清单.md**
- 按品类给出典型调性定义：
  - 科技类品牌：避免感叹号、避免情感词、用数据和成分说话
  - 母婴类品牌：避免夸张、避免绝对承诺、突出安全认证
  - 美妆类品牌：避免医疗化表述、避免同行比较、强调感受和体验
- 每类给出"允许的词汇范围"和"禁用的表达方式"

**knowledge/raw/rules/事实核查要点.md**
- 数据引用必须标注来源
- 成分功效必须有依据
- 不能编造用户评价
- 不能虚构使用效果

**验收标准**：
- 四个规则文件创建完毕，每个文件内容 > 500 字
- search_knowledge 能搜到这些规则（关键词如"禁用词""小红书字数"）

**预估耗时**: 2-3h（内容撰写）

### T2-02: 更新 reviewer/soul.md — 规则驱动审核流程

**目标**: 审核 Agent 必须先查规则再审，审核理由引用规则来源

**产出**: 修改 `agents/reviewer/soul.md`

**body 关键指令添加**：
```markdown
## 审核工作流

对每一条内容：

1. 调 `search_knowledge` 查询相关规则：
   - 搜 "禁用词 [品类]" 查广告法
   - 搜 "[平台] 规范" 查平台要求
   - 搜 "[品牌调性] 检查" 查调性要求
   
2. 逐条对照规则审查成稿，给每条不通过的内容标注：
   - 违反的具体规则（引用规则文件）
   - 问题出现的位置（引用成稿原文）
   - 修改建议

3. 审核反馈格式：
   ```
   ❌ 不通过
   
   【违规点1】使用了"最有效"（违反：广告法禁用词.md 第X条）
   出现位置："...是市面上最有效的精华液..."
   修改建议：改为"表现优异的"或直接删除修饰语
   
   【违规点2】...
   ```

**禁止给出"语言不够自然"这种主观模糊的反馈**。所有驳回必须有规则依据。
```

**验收标准**：
- 审核 Agent 的 review_feedback 字段能看到规则引用
- 不再出现"感觉不太对"这种主观描述

**依赖**: T2-01

**预估耗时**: 30min

### T2-03: 审核经验沉淀的 Hook prompt 定制

**目标**: 审核发现的违规模式写入 wiki，反哺文案 Agent

**产出**: 修改 `agents/base.py` 的 _hook_reflect，审核角色专用 prompt

**审核专用自省 prompt**：
```
你刚完成了一轮审核工作。回顾这次发现的问题：

1. 这次驳回了哪些内容？各自违反了什么规则？
2. 这些问题本可以在文案撰写阶段避免吗？
3. 归纳出一条"文案 Agent 下次撰写此类内容时应该预先检查的要点"

输出 JSON:
{
  "situation": "某品类的文案审核",
  "violations_found": ["发现的违规类型1", "违规类型2"],
  "action": "审核策略",
  "outcome": "通过率 X%",
  "lesson": "下次撰写 [品类] 的 [平台] 内容时，必须预先检查：...",
  "category": "电商大促|新品发布|品牌传播|日常运营",
  "applicable_roles": ["reviewer", "copywriter"]   # 注意这里要包含 copywriter
}
```

**关键点**: `applicable_roles` 包含 `copywriter`，这样文案 Agent 下次启动时会自动注入这些经验。

**验收标准**：
- 审核经验卡片的 applicable_roles 字段包含 copywriter
- 文案 Agent 下次运行时 system prompt 中能看到这些审核经验

**依赖**: T2-02, 阶段六的 Hook 机制

**预估耗时**: 1h

### T2-04: 验证"审核经验反哺文案"的闭环

**目标**: 测试证明文案 Agent 真的从审核经验中学到了东西

**产出**: `tests/test_reviewer_feedback_loop.py`

**测试流程**：
1. 清空经验池
2. 跑第一个项目（测试 Brief A）
3. 检查审核 Agent 蒸馏的经验：是否包含具体违规模式
4. 跑第二个项目（测试 Brief B，同品类）
5. 检查第二次文案 Agent 的 system prompt：是否包含第一次的审核经验
6. 对比两次的审核通过率：第二次应该显著更高

**期望结果**：
```
第一次运行：审核通过率 60%（3/5 条通过）
  主要违规：使用绝对化用语 2 次
  
经验沉淀：applicable_roles=[reviewer, copywriter]

第二次运行：审核通过率 100%（5/5 条通过）
  未出现第一次的违规类型
  
闭环验证：✅ 审核经验成功反哺文案创作
```

**依赖**: T2-01, T2-02, T2-03

**预估耗时**: 1-2h

## 模块三：客户经理 Agent 的"人类指导进化"

### T3-01: 实现 request_human_review 工具

**目标**: 客户经理生成 Brief 解读后，通过 IM 发给人类审核，阻塞等待回复

**产出**: `tools/request_human_review.py`

**方案**：
- 用飞书卡片的交互回传事件
- 卡片带"通过"/"修改"按钮，人类点击后触发 webhook
- Agent 通过事件等待机制获取结果
- 实现更复杂，不推荐比赛用

**SCHEMA**:
```json
{
  "name": "request_human_review",
  "description": "将 Brief 解读提交给人类专家审核，获取修改建议",
  "parameters": {
    "brief_analysis": "你生成的 Brief 解读全文",
    "timeout_seconds": 300
  }
}
```

**返回格式**：
```
人类审核结果：
状态：需要修改
修改意见：客户说的年轻化实际指 22-28 岁，不是 25-35 岁。另外客户之前提过不要工业风设计感。

请根据以上意见重新生成 Brief 解读。
```

或：
```
人类审核结果：
状态：通过
意见：解读准确，可以进入下一阶段。
```

**关键设计**：
- Demo 模式开关 `AUTO_APPROVE_HUMAN_REVIEW` 环境变量，为 true 时跳过人类审核直接通过（但打印"模拟人类批准"日志）
- 这个开关让 Demo 既可以演示人机协同（手动回复），也可以快速跑通（自动通过）

**验收标准**：
- 工具注册到 ToolRegistry
- 真实发送 IM 消息到群聊
- 能正确解析 "通过" / "修改：xxx" 两种回复
- 超时机制工作正常
- AUTO_APPROVE 模式可以跳过

**预估耗时**: 3-4h

### T3-02: 更新 account_manager/soul.md — 强制人类审核流程

**目标**: 客户经理必须走人类审核，不能跳过

**产出**: 修改 `agents/account_manager/soul.md`

**tools 新增**:
```yaml
tools:
  - read_project
  - write_project
  - update_status
  - request_human_review    # 新增
  - send_message
```

**body 关键指令**：
```markdown
## 强制工作流

1. 读取 Brief 原文和已有上下文
2. 生成初版 Brief 解读
3. **必须**调用 `request_human_review` 提交给人类审核
4. 根据人类反馈：
   - 通过 → 写入 brief_analysis 字段，更新状态
   - 需修改 → 根据意见重新生成，再次提交审核（最多 2 轮）
5. 在最终 brief_analysis 中明确标注"已通过人类审核"以及采纳了哪些人类建议

**不经人类审核直接输出视为违规**。

## 经验沉淀侧重点

当人类给出修改意见时，分析：
- 人类看到了什么你没看到的？
- 人类的判断依据是什么？
- 下次遇到类似客户时，这条经验如何应用？
```

**验收标准**：
- 客户经理在 ReAct 中必定出现 request_human_review 调用
- 修改后的 Brief 解读明显吸收了人类意见

**依赖**: T3-01

**预估耗时**: 30min

**验收标准**：
- 经验 JSON 包含 human_correction 和 reasoning 字段

### T3-04: Demo 现场人机协同脚本

**目标**: Demo 时有可操作的人机协同流程，不会卡死

**产出**: `demo/human_review_playbook.md`

**内容**：


**验收标准**：
- 至少覆盖 Demo 会用到的每条测试 Brief
- 有队友演练过至少 1 次

**预估耗时**: 1h

## 模块四：演示进化效果的对比测试

### T4-01: 实现人机干预次数追踪

**目标**: Demo 时能展示"第一次 vs 第五次"的量化对比

**产出**: `memory/evolution_metrics.py`

**追踪指标**：
- 每次项目运行的 human_review 调用次数
- 每次项目的审核驳回条数
- 每次项目的 search_reference 命中次数
- 每次项目的 search_knowledge 命中次数

**存储方式**: 简单 JSON 文件 `evolution_log.json`，记录每次项目的统计数据

**查询 API**:
```python
get_metrics(project_type="电商大促") 
→ [
    {"run_n": 1, "human_corrections": 5, "review_rejections": 4, ...},
    {"run_n": 2, "human_corrections": 3, "review_rejections": 2, ...},
    ...
]
```

**验收标准**：
- 每次项目运行后自动记录指标
- 能查询出某品类的历次运行数据

**预估耗时**: 1-2h

## 时间预算

总计约 18-25 小时工作量。按三人分工：
- 一人做模块一（文案对标，4-5h）
- 一人做模块二（审核规则，5-7h）
- 一人做模块三（人类审核，5-7h）
- 最后共同做模块四（集成+Demo，4-6h）

两天可以全部完成。

## 什么不做

- 不做复杂的人类审核事件驱动架构（用简单 poll 够了）
- 不做真实的小红书爬虫（SerpAPI 或本地 mock 就行）
- 不做规则的自动学习（规则是人写死的，这是"客观权威"的意义）
- 不做多轮人类审核（最多 2 轮，防止 Demo 死循环）

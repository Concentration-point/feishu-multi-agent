# IMAGE-GENERATION-SOP.md

## 目标

把外部 GPT-Image-2 + multi-agent 图像工作流内化成 Clawd 自己的生图流程：少废话、先拆解、直接产出、失败就换路，不照搬任何未审计 skill。

## 适用场景

- 个人形象图卡 / 证件照 / 海报 / UI mockup / 产品图 / 系列图
- 用户要求“参考某工作流”“学习某案例”“用 gpt_image_2 生成”
- 需要多张图保持风格统一、角色一致或版式一致
- Feishu 用户明确说“生图 / 生成图片 / 绘制 / 图卡 / 海报 / 证件照 / 职业照 / 头像 / 形象分析图 / 检测生图链路”时，直接进入 `image_generate`，不空回复、不只回 `Done.`、不只写 prompt。

## 强制执行原则

- 生图不是“写一个长 prompt”。每次高要求视觉任务都必须实际跑闭环：**Brief → Prompt → Generate → Critique → Regenerate if needed**。
- 可以不把内部 brief 全部发给用户，但不能跳过。
- 如果图内中文不是任务核心，默认少字/无字，避免模型乱排中文；准确文字单独给后期叠加。
- 用户说“更漂亮一点 / 高级一点 / 不够像 / 继续”时，不许只加形容词重抽；必须基于上一版缺陷换构图、色彩、主视觉或版式策略。
- 出图后若明显不合格，直接重试；不要用“生成好了”糊弄。

## Clawd 版工作流

### 1. 意图翻译

把用户的大白话转成设计 brief：

- 主体：画谁/什么
- 用途：头像、海报、封面、图卡、UI、产品图
- 风格：写实、插画、电影感、极简、科技感、国风等
- 约束：比例、文字、是否保留脸、是否参考上传图
- 禁区：不要像真人明星、不要过度修图、不要乱改身份特征

### 2. 任务拆解

复杂图不要一口吃：

- 先确定构图和信息层级
- 再确定主视觉
- 再确定色彩/材质/光影
- 最后处理文字和版式

对“图卡/分析卡”类，先拆字段再生成：标题、分区、标签、对比项、结论。

### 3. Prompt 编译

用 Clawd 自己的风格：短、准、可控，不写论文式 prompt。

模板：

```text
原创图像：[主体]，[场景/构图]，[风格]，[关键细节]，[色彩/光影]，[质量要求]，[负面限制]。
```

人像类必须加：

```text
保留原始五官、脸型、肤色和真实人类质感，不要网红脸，不像任何真实名人。
```


### 3.1 Prompt Fidelity Protocol（用户已有 prompt 时）

当老大已经给出明确生图 prompt，不允许把它“重写成另一个需求”。我的职责是**保真增强**，不是夺舍。

执行顺序：

1. **原始意图锁定**：先提取用户 prompt 里的主体、用途、风格、比例、文字、参考图、禁区、必须包含项。
2. **硬约束不可删**：用户明确写出的字段、分类、对比关系、风格词、限制词，默认全部保留；不能因为我觉得“更好看”就改掉。
3. **软增强只补洞**：只补模型执行需要的构图、光线、材质、清晰度、负面限制；不替换用户核心审美。
4. **冲突先显式处理**：如果用户 prompt 内部冲突，或图像模型明显做不到，先指出冲突并选择最保守方案；不要偷偷改。
5. **迭代按失败点修**：用户不满意时，必须基于上一版缺陷定向修改：构图、色彩、信息层级、文字策略、主体一致性；禁止完全另写一个“看起来高级”的新 prompt。

输出给工具的 prompt 应采用：

```text
保留用户原始需求：[逐项保留]
结构化补强：[构图/光线/质感/负面限制]
不可违反：[用户硬约束]
```

如果用户 prompt 已经足够明确：少改，只做清晰化和执行化。
### 4. 生成执行

- Feishu 明确生图请求必须调用 `image_generate`，模型写完整 `openai/gpt-image-2`；当前或最近上下文有上传图路径时，必须传入 reference image。
- Feishu 会话里使用 `image_generate` 后，必须发可见短 caption：`生成好了。`
- 指定模型优先写完整：`openai/gpt-image-2`，避免默认 provider 掉到 custom-api。
- OpenAI 图像模型不支持 `aspectRatio` 时改用 `size`。
- 失败必须说明失败点并马上换参数/模型名重试，不准空消息。

### 5. 质量审查

出图后按任务检查：

- 是否满足比例/用途
- 是否保留主体特征
- 是否有乱字/错字
- 是否风格统一
- 是否过度修图
- 是否违反用户明确要求
- 是否只是“好看词堆砌”而没有解决用户要的功能
- 如果是海报：是否有官方感、层级、留白、主视觉、准确文字策略

必要时直接二次生成，不要把明显烂图硬说“可以”。

### 5.1 迭代规则

用户要求“再漂亮一点 / 更高级 / 继续优化”时，必须至少改变一个核心设计变量：

- 构图：中心构图、留白、透视、前中后景
- 主视觉：校园、人物、产品、符号、场景
- 色彩：主色、辅色、光影、材质
- 信息策略：少字/无字、标题区、后期叠字区
- 质感：摄影、3D、插画、数字艺术、官方海报

禁止只在 prompt 里追加“更漂亮、更高级、4K”。这很废。

### 6. 系列化/批量图

如果用户要一套图：

- 先定统一视觉规范：色板、背景、镜头、字体感、构图
- 每张图只改变变量：发型/妆容/珠宝/颜色/职业等
- 保持同一主体、同一光线、同一版式

### 7. 案例库复用

成功 prompt 和失败点写入记忆或 daily note；下次复用结构，不复制原文。

## 与老大偏好的结合

- 默认中文简短汇报，不写长篇解释
- 能直接生成就直接生成
- 工具失败不要沉默，直接说失败原因和下一步
- 老大要“检测链路”时，生成明确测试卡
- 老大要“印象图/职业照”时，允许带 Clawd 的龙虾符号，但别喧宾夺主

## 硬边界

- 不生成冒充证件/官方文书的欺骗性材料
- 不照搬未审计外部 skill 或安装新依赖
- 不在群聊泄露私人画像、记忆或偏好
- 不对真人照片做过度修图或身份误导

# PPT-GENERATION-SOP.md

> 目标：把“让 AI 做 PPT”从玄学变成稳定流水线。
>
> 核心判断：AI 生成 PPT 的本质不是直接操作 PowerPoint，而是把内容、叙事、审美系统翻译成可执行脚本，生成一份可编辑 `.pptx`。

## 0. 总原则

```text
优先新生，不优先改旧。
优先原生生成，不优先微调模板。
优先可编辑 PPT，不优先图片糊成整页。
优先先定叙事和审美，再写代码。
```

AI 最擅长的是从干净材料生成新 deck；最容易翻车的是“帮我把这个人类做好的复杂 PPT 改得更好”。

## 1. 场景分流

### A. 从 0 生成完整 PPT —— 默认推荐

适合：
- 课堂汇报
- 论文汇报
- 商务汇报
- 项目路演
- 课程展示
- 主题分享

输入：
```text
文字大纲 / 论文 / 文档 / 截图 / 数据 / 参考风格 / 页数 / 时长 / 听众
```

输出：
```text
可编辑 .pptx
可选 PDF 预览
可选每页预览图
可选图片素材包
```

### B. 给已有 PPT 新增几页 —— 可以做

适合：
- 小组 PPT 中补自己负责的几页
- 已有 deck 只是作为上下文和风格参考
- 在原 PPT 后插入新章节

规则：
```text
只新增，不大改。
先分析旧 PPT 风格，再生成新页。
新页尽量匹配旧 PPT，但不要破坏旧页。
```

### C. 修改旧 PPT —— 谨慎

不推荐大改人类做好的旧模板。

原因：
```text
旧 PPT → AI 解析 → 转成代码理解 → 修改 → 再生成 PPT
```

这中间有损耗，母版、字体、图层、动画、精细间距都可能坏。

允许：
- 改文字
- 替换图片
- 补少量页面
- 统一少量格式
- 修改 AI 自己之前生成的 PPT

禁止默认承诺：
- 大规模美化旧 PPT
- 无损复刻复杂模板
- 保留全部动画和母版
- 像人类设计师一样局部精修

## 2. 输入准备清单

生成前必须尽量收齐：

```text
1. 主题：这份 PPT 要讲什么
2. 场景：课堂 / 答辩 / 商务 / 小组 / 作品集
3. 听众：老师 / 同学 / 领导 / 客户 / 评委
4. 时长：几分钟
5. 页数：大概几页
6. 原材料：论文、文档、大纲、数据、截图
7. 风格参考：模板图、网页、已有 PPT、skill 风格说明
8. 硬约束：必须包含什么，不能出现什么
9. 输出要求：PPT / PDF / 图片素材 / 演讲稿
```

缺少输入时，允许先用合理默认值推进，但必须标注假设。

## 3. 标准流程

### Step 1：材料消化

先不要做 PPT。先让 AI 理解材料。

产物：
```text
- 核心主题
- 受众判断
- 信息优先级
- 讲述主线
- 必须出现的概念 / 数据 / 图表
- 应该压缩或删除的内容
```

提示词：
```text
请先不要制作 PPT。请阅读下面材料，提炼：
1. 这份 PPT 的核心观点
2. 目标听众最需要听懂什么
3. 最适合的叙事主线
4. 哪些内容必须保留
5. 哪些内容应该压缩或删除
6. 建议页数和章节结构
```

### Step 2：生成叙事架构 Storyboard

不出 Storyboard，不写脚本。

每页格式：
```text
第 N 页：页面标题
页面目的：
核心文案：
视觉方案：
备注：
```

要求：
```text
每页只讲一个意思。
每页都要有存在理由。
先让结构成立，再让页面好看。
```

### Step 3：提取审美系统

审美必须外部注入。不要信模型默认审美。

可用来源：
- zarazhang / frontend-slides 这类 HTML 风格 skill
- 小红书 / GitHub / 网页参考
- PPT 截图
- 品牌手册
- 老大给的模板图

#### 使用 HTML skill 的正确方式

如果 skill 是为 HTML 幻灯片设计的，不要直接执行 HTML。

正确链路：
```text
HTML skill / frontend-slides → 设计系统 → PPT 代码生成
```

错误链路：
```text
HTML skill → 直接生成 HTML → 硬转 PPT
```

让 AI 只输出设计系统：
```text
请不要生成 HTML，也不要执行代码。请只根据这个 skill 的风格，提取一套适合 PPT 生成的设计系统：字体、色号、版式、留白、组件、封面页、章节页、内容页、图表页规则。
```

#### 使用参考图的方式

要求 AI 只参考：
```text
颜色
留白
字体气质
版式密度
图形语言
```

不要照抄具体内容。

提示词：
```text
请不要制作 PPT。请只分析这组参考图的视觉风格，并输出一套可用于 python-pptx 生成的设计系统，包括：
1. 主色 / 辅色 / 背景色 / 强调色，给出 HEX
2. 中文字体 / 英文字体建议
3. 标题字号、正文字号、注释字号
4. 页面边距、栅格、留白规则
5. 封面页、章节页、内容页、图表页、结论页的版式规则
6. 禁止事项：哪些风格不要出现
```

### Step 4：生成 PPT 制作 Prompt

正式生成前，先产出 `PPT_PRODUCTION_PROMPT.md`。

必须包含：
```text
- 项目目标
- 目标听众
- 页数与时长
- 章节结构
- 每页 Storyboard
- 设计系统
- 图片策略
- 代码技术栈
- 可编辑性要求
- 验收标准
```

这一步的价值：拉长有效上下文，让模型先完整规划，再进入执行。

### Step 5：生成可编辑 PPT 脚本

推荐技术栈：
```text
python-pptx
```

复杂视觉也可考虑：
```text
pptxgenjs
```

硬要求：
```text
1. 标题、正文、标签、页码必须是 PPT 文本框，可编辑
2. 中文不要烘焙进图片
3. 图形尽量用 shape / line / table / chart
4. 图片只放装饰图、示意图、截图、插画
5. 每页统一 safe margin
6. 不允许文本溢出
7. 不允许字体随机漂移
8. 不允许每页风格重开一局
9. 脚本必须能重复运行
10. 输出路径固定、可追踪
```

### Step 6：图片素材单独处理

图片不要和 PPT 主结构一锅炖。

正确流程：
```text
先生成无图 / 占位图 PPT
再列出每页需要什么图片
再逐张生成图片
最后插入 PPT
```

图片 prompt 单独保存：
```text
slide_03_visual_prompt.txt
slide_07_visual_prompt.txt
```

图中文字规则：
```text
中文尽量不进图。
如果必须进图，只放极短标签。
复杂中文统一放 PPT 文本框。
```

### Step 7：自动验收

生成后必须检查，不准只看“脚本运行成功”。

验收项：
```text
1. .pptx 文件存在且可打开
2. 页数正确
3. 每页标题存在
4. 没有空白页
5. 文本没有明显溢出
6. 字体统一
7. 色号符合设计系统
8. 图片路径有效
9. 没有公式乱码
10. 导出 PDF 后视觉正常
11. 每页预览图没有明显错位
```

推荐产物：
```text
output.pptx
output.pdf
previews/slide_01.png ...
```

### Step 8：迭代规则

按问题类型回退，不要乱改。

```text
结构问题 → 回到 Storyboard
风格问题 → 回到 Design System
单页问题 → 重生成单页
图片问题 → 只重生成图片
文案问题 → 局部替换文本
```

禁止：
```text
让 AI 对人类精修过的旧模板做大规模微调。
```

允许：
```text
让 AI 对自己生成的 PPT 继续改。
让 AI 给已有 PPT 新增几页。
让 AI 根据已有 PPT 风格重做一套。
```

## 4. 论文 / 课堂汇报专用分支

### 4.1 论文材料处理

输入论文后，先转成结构：
```text
研究背景
问题定义
方法/模型
实验设置
关键结果
贡献点
局限性
课堂讲述重点
```

不要把论文原文塞进 PPT。

### 4.2 推荐页序

```text
1. 标题页：论文题目 + 汇报人
2. 研究背景：为什么这个问题重要
3. 核心问题：论文到底想解决什么
4. 方法总览：一张图讲清 pipeline
5. 关键机制 1：只讲最重要的模块
6. 关键机制 2：必要时展开
7. 实验设置：数据集 / 指标 / 对比方法
8. 结果分析：核心表格或图，不堆所有结果
9. 优点与局限：客观评价
10. 总结：三句话收束
```

可按汇报时长增删。

### 4.3 公式规则

论文 PPT 最容易死在公式上。

规则：
```text
1. 只放核心公式
2. 每页最多解释一个公式重点
3. 复杂公式可转高清透明 PNG
4. 公式旁边必须有可编辑中文解释
5. 不要整页堆推导
6. 不默认承诺公式完全可编辑
```

如果必须可编辑公式，再考虑：
```text
LaTeX → MathML / OMML → Office Equation
```

但这是额外工程，不默认启用。

### 4.4 图表规则

```text
结果表格只保留关键列。
趋势图优于密集表格。
每张图必须配一句 takeaway。
不要把论文截图直接糊满一页。
```

## 5. 小组 PPT 新增页分支

适合用户只负责几页。

流程：
```text
1. 读取或截图分析现有 PPT 风格
2. 提取字体、颜色、页边距、标题样式、组件风格
3. 只生成新增页面
4. 新页风格贴近旧 PPT
5. 不改旧页，除非用户明确要求
```

提示词：
```text
这是已有小组 PPT 的参考截图 / 文件。请只分析它的视觉风格，并为我新增 N 页内容。不要修改旧页面。新增页面需要尽量匹配原 PPT 的字体、颜色、边距、标题样式和信息密度。
```

## 6. 直接可用总 Prompt

```text
你现在是 PPT 策划、视觉设计师和 python-pptx 工程师。

请严格按流程工作，不要一上来生成 PPT。

任务：基于我提供的材料，从 0 原生生成一份可编辑 PPT。

要求：
1. 先理解材料，提炼核心观点、听众、时长、页数建议。
2. 再输出完整 Storyboard，每页说明：标题、目的、核心内容、视觉方案、备注。
3. 再根据我提供的风格参考 / skill 风格，提取 PPT 设计系统，包括字体、色号、版式、间距、组件风格。
4. 再生成一份 PPT 制作 Prompt，整合内容结构、视觉系统、代码要求和验收标准。
5. 最后写 python-pptx 脚本生成 .pptx。
6. PPT 中所有标题、正文、标签、页码必须可编辑；不要把中文文字烘焙进图片。
7. 图片先用占位图或单独生成，不要影响 PPT 主结构。
8. 生成后必须检查：页数、空白页、文本溢出、字体统一、公式乱码、风格一致性。
9. 如果需要修改，优先重生成单页或重建整套，不要硬改旧模板。

现在请先执行第 1 步：理解材料并输出 PPT 叙事架构，不要写代码。
```

## 7. Codex / Claude Code / GPT 使用建议

### Codex / Claude Code

适合：
```text
生成脚本
调试 python-pptx
批量插图
导出预览
修复依赖
自动验收
```

### 普通 GPT / Claude 聊天

适合：
```text
材料理解
叙事架构
设计系统
生成 prompt
少量页面文案
```

### 最佳组合

```text
聊天模型：做理解、架构、审美方案、制作 prompt
Coding agent：写脚本、生成文件、导出预览、自检修复
Image model：逐张生成视觉素材
人类：最终审美判断和少量文字精修
```

## 8. 失败模式与纠偏

### 页面太空

原因：内容密度规划不足。

处理：
```text
回到 Storyboard，补充每页信息层级：主标题 / 副标题 / 3 个要点 / 图示 / takeaway。
```

### 页面太满

原因：把讲稿塞进 PPT。

处理：
```text
每页只保留一个观点，细节放备注或讲稿。
```

### 设计丑

原因：缺审美参考。

处理：
```text
补参考图或提取 frontend-slides / PPT 模板风格系统。
```

### 公式乱码

原因：字体 / LaTeX / Office 公式链不稳定。

处理：
```text
核心公式转高清透明 PNG + 可编辑中文解释。
```

### 图片不稳定

原因：生图和 PPT 同时生成，链路耦合。

处理：
```text
图片单独生成，PPT 先用占位。
```

### 改旧 PPT 越改越坏

原因：解析旧 PPT 有损。

处理：
```text
停止微调旧文件，改为提取风格后重做，或只新增页面。
```

## 9. 必须遵守的铁规

```text
1. 不从旧 PPT 大规模微调开始。
2. 不跳过 Storyboard。
3. 不跳过设计系统。
4. 不把大量中文烘焙进图片。
5. 不把图片生成和 PPT 主结构绑定死。
6. 不只看脚本成功，必须看导出/预览。
7. 不承诺无损复刻复杂模板。
8. 不把论文原文原封不动塞进页面。
9. 不让每页风格随机漂移。
10. 不把“可编辑 PPT”做成一堆截图。
```

## 10. 最小交付标准

一次正式 PPT 生成任务至少交付：

```text
1. .pptx
2. 生成脚本
3. Storyboard 或页面清单
4. 设计系统摘要
5. 验收结果
```

高要求任务额外交付：

```text
1. PDF 预览
2. 每页 PNG 预览
3. 图片 prompt 清单
4. 素材目录
5. 修改建议
```

# 检索与信息处理 SOP

## 目标
避免单一工具依赖。以**拿到结果**为第一原则，而不是执着某一个搜索/浏览工具。

## 核心原则
1. **先判断信息类型，再选链路**。
2. **一个工具失败，立刻切备用链路**，不要原地卡住。
3. **默认做交叉验证**：原始来源 + 第二来源；必要时加实测。
4. **优先原始来源**：官方文档、原文页面、源码、公告。
5. **输出要重组**：不是把抓到的内容原样倒给用户。

## 默认工作流
### A. 普通网页 / 新闻 / 博客 / 攻略
1. `web_search` 找入口
2. `web_fetch` 抓正文
3. 若正文缺失/动态渲染失败：改用 `browser`
4. 若 `browser` 不可用：改用 `exec + 本地浏览器/CLI`
5. 至少做一次第二来源核对

### B. 动态页面 / 登录态 / 复杂交互
1. 优先 `browser`
2. 若 `browser` 故障：
   - `exec` 调本地 Chrome / Edge headless
   - 必要时走页面打印、导出、截图、PDF
3. 不在浏览器工具失败后停住

### C. 文档类（Notion / 飞书 / 云文档 / PDF）
1. 优先平台直链抓取或专用工具
2. 如果要交付 PDF：
   - 先尝试原生导出
   - 不行就用 headless 浏览器打印 PDF
3. 若用户要求“包括图片”，优先走渲染/打印链路，不要只抓纯文本

### D. 本地文档 / 配置 / 代码
1. `read`
2. 大范围检索用 `exec`（如 grep / rg / Select-String）
3. 需要验证行为时，优先做本地实际测试

## 故障切换顺序
当某条链路失败时，按这个顺序切：
1. `web_search`
2. `web_fetch`
3. `browser`
4. `exec + 本地浏览器（Chrome / Edge headless）`
5. 平台专用工具
6. 本地脚本重排 / 导出

## 质量标准
### 默认至少两层验证的场景
- 教程步骤
- 配置方法
- 产品功能
- 时效性新闻
- 攻略 / 路线 / 推荐

### 验证方式
- **源 A：原始来源**
- **源 B：二次验证**（社区、镜像文档、实测）
- **源 C：必要时实际操作验证**

## 输出规范
1. 先给结论
2. 再给关键依据
3. 再给操作建议 / 下一步
4. 用户要求整理版时，做结构化重写，不直接贴采集原文

## 常用执行模板
### 网页信息整理
- 搜索入口
- 抓原文
- 二次核对
- 提炼重点
- 输出结论 + 风险 + 下一步

### 晨报 / 热榜检索
- 遵循 `MORNING-REPORT-SOP.md`
- 对热榜页、强交互页、易反爬页、需要真实排序的页：真实浏览器优先
- 原始目标页优先，其次聚合页，再其次搜索恢复，最后新闻流反推
- 已还原历史有效链：知乎热榜可走官方页 `https://www.zhihu.com/hot`；微博热搜历史稳定证据优先指向 tophub 聚合页
- 热榜原榜抓不到时必须标注“搜索恢复版”或“人话替代版”
- 需要页面顺序、版式、截图时，优先浏览器链或本地 headless 浏览器

### 网页转 PDF（含图）
- 优先原生导出
- 失败则 headless Chrome/Edge `print-to-pdf`
- 检查文件生成是否成功
- 再发送，不要先承诺

## 明确禁止
- 因为某个工具失败就停止推进
- 把“没抓到”当成“没有”
- 在需要图片时只导出纯文本
- 没核对就把教程步骤当真

## 一句话版本
**工具只是爪子，不是脑子。目标是拿到可靠结果。**
